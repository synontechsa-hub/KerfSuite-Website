# Codebase Audit Report — KerfPortal

**Date:** 2026-07-20
**Agent:** SoulLink Agent (Gemini)
**Scope:** Architecture, Security, Stability, and Edge Cases.

---

## 1. Executive Summary

The codebase is well-structured and follows modern Next.js + Supabase patterns. It demonstrates a strong focus on security (RLS, MFA, HMAC leases) and domain separation. However, there are critical race conditions in the inventory system and some remaining security risks that contradict the "hash-only" storage philosophy for secrets.

---

## 2. Critical & High Priority Issues

### 🔴 Race Condition: Sequential Asset Naming
**File:** `src/app/api/stock/assets/route.ts` & `src/app/api/stock/kerfcut/commit/route.ts`
- **Issue:** The `system_name` (e.g., `SHEET-0001`) is generated by counting existing rows in a `GET` request and adding 1 in code.
- **Risk:** Two concurrent requests will generate the same `system_name`. If business logic or physical labels rely on this being unique, it will cause data corruption and physical tracking errors.
- **Fix:** Move the naming logic into a PostgreSQL trigger or use a `SEQUENCE` per workspace/type.

### 🔴 Non-Atomic "Commit" Logic
**File:** `src/app/api/stock/kerfcut/commit/route.ts`
- **Issue:** The API performs multiple independent DB calls (Update assets, Insert remnants, Insert events) and uses a manual "rollback" if the first update is partial.
- **Risk:** If the process crashes or subsequent calls fail, the database is left in a "half-committed" state (e.g., sheets consumed but remnants not created).
- **Fix:** Wrap the entire commit logic in a single PostgreSQL RPC function to ensure ACID compliance.

### 🟠 CDKey Leakage in Revocation
**File:** `src/app/portal/actions.ts` (`revokeKey`)
- **Issue:** The `revokeKey` action logs the raw `cdkey` returned from `PortalService.revokeLicense`.
- **Risk:** While `generateKey` is now hardened to only log suffixes, `revokeKey` still exposes the full key in `audit_logs`, which are viewable by all workspace members.
- **Fix:** Mask the key in the revocation audit log.

### 🟠 Cross-Workspace Location Parenting
**File:** `src/app/api/stock/locations/route.ts`
- **Issue:** When creating a location with a `parent_id`, the code fetches the parent's depth without verifying the parent belongs to the user's `workspace_id`.
- **Risk:** A user could potentially guess/obtain a UUID of a location in another workspace and link their inventory to it, potentially causing RLS side-effects or data leakage if queries aren't perfectly scoped.
- **Fix:** Ensure `parent_id` belongs to the same `workspace_id` before proceeding.

## 3. Architectural & Maintainability Flaws

### 🟡 Service Layer Inconsistency
- **Finding:** `PortalService` is used for many operations, but several API routes (especially under `/api/stock`) still perform direct `supabase` calls.
- **Impact:** Fragmented business logic makes it harder to apply global changes (like the naming fix mentioned above).
- **Recommendation:** Move all inventory management logic into `PortalService`.

### 🟡 Lack of Strict DB Typing
- **Finding:** Mappers in `src/models/portal.ts` use `any` for DB inputs.
- **Impact:** Loss of compile-time safety when the schema changes.
- **Recommendation:** Generate TypeScript types from the Supabase schema and use them in the mappers.

---

## 4. Security Enhancements

### 🟢 Timing Attacks on Provisioning
**File:** `src/app/api/provision/route.ts`
- **Finding:** The `PROVISIONING_SECRET` is checked using standard string comparison.
- **Recommendation:** Use `crypto.timingSafeEqual`.

### 🟢 CDKey Storage Enforcement
- **Finding:** Comments suggest CDKeys should never be stored raw, but the schema doesn't enforce this.
- **Recommendation:** Add a CHECK constraint to the `license_slots` table if the goal is to purge them after generation.

---

## 5. Positive Observations
- **MFA Enforcement:** Very strong implementation of `ensureAdminWithMFA`.
- **Lease Model:** The offline lease token generation is robust and handles environment failures correctly.
- **UI Architecture:** Clean separation of Client and Server components.
